Unifying fault sources

The system had two distinct fault origins ‚Äì customer‚Äëreported faults and maintenance‚Äëvisit failures. Technicians needed a single dashboard to see both types without confusion.

Solution:

Designed a combined view model with dynamic filtering by source and status. Used Discriminator pattern in Entity Framework to preserve audit trails while presenting a unified technician experience.

üí£

Replacement integrity

When a technician marked a fridge as ‚Äúscrapped‚Äù, the system had to atomically: update fault status, create a replacement request, notify customer support, and reserve a new fridge ‚Äì all or nothing.

Solution:

Implemented explicit database transactions with TransactionScope across multiple DbContext operations. Added compensating logic for approval/rejection, ensuring inventory never desynchronized.

üîí

Support role isolation

Business rules required that Customer Support could approve/decline replacement requests without viewing fault details (to maintain objectivity). Yet they needed to know who requested it and why.

Solution:

Created a projection view model that strips fault details, showing only replacement summary. Implemented separate authorization filters to block direct navigation to fault reports from the support interface.

‚õî

State transition rules

Fault reports could only follow 12 valid paths (e.g., Reported ‚Üí Scheduled ‚Üí InProgress ‚Üí Resolved). Buttons had to be conditionally shown based on both role and current status.

Solution:

Built a state‚Äëtransition matrix in the view model, with Razor logic that renders buttons only when IsValidTransition(current, target) returns true. Eliminated all invalid status updates at the UI level.

‚è∞

Auto‚Äëclose after 7 days

Business rule: resolved faults should auto‚Äëclose after 7 days if not manually closed. Needed a non‚Äëintrusive background mechanism.

Solution:

Added an AutoCloseFaultIfNeeded method called during any fault view access. For scrapped faults, closure is immediate; for resolved faults, it checks ResolvedDate.AddDays(7) <= DateTime.Now.

üí¨

Internal vs. external comments

Technicians needed a way to leave internal notes (hidden from customers) alongside public updates.

Solution:

Added IsInternalNote boolean to FaultComment. Customer views filter out internal notes; technician views show them with distinct styling. Used the same property to generate two‚Äëcolor timelines.
